# v0.19.0 — dict-uniform API for the four remaining positional methods

**Theme: API uniformity.** Every multi-arg public method now takes a single
dict, matching the shape that `create_video` / `create_image` / `edit_image`
established in v0.18.0. The old positional/kwargs forms still work but emit
`DeprecationWarning` and will be removed in **v0.20.0**.

## Why

`create_video({...})` and `extend_video(video_id, ..., **kwargs)` having
different shapes was a footgun for LLM-driven callers — muscle memory from
the dict-style methods caused dict-swallow bugs (the dict landed as
`video_id=<dict>` and was silently accepted). It also made the CLI / Python
SSOT (per the cli-steering-engineering doc) lopsided: half the API was
schema-driven, half wasn't.

## What changed

Four methods became dict-style:

| Method | Old | New |
|---|---|---|
| `extend_video` | `extend_video(video_id, *, prompt, duration, seed_start, timeout, preserve_source_favorite_state)` | `extend_video({"video_id": ..., "prompt": ..., ...})` |
| `download_video` | `download_video(video_id, output_path, *, prefer_hd, parent_post_id)` | `download_video({"video_id": ..., "output_path": ..., "prefer_hd": ...})` |
| `select_post` | `select_post(post_id, *, timeout)` | `select_post({"post_id": ..., "timeout": ...})` |
| `wait_for_video_completion` | `wait_for_video_completion(video_id, timeout, poll_interval)` | `wait_for_video_completion({"video_id": ..., "timeout": ..., "poll_interval": ...})` |

Internally the schema gained three new PARAMS entries (`output_path`,
`prefer_hd`, `poll_interval`) and three new KEY lists (`DOWNLOAD_KEYS`,
`SELECT_POST_KEYS`, `WAIT_FOR_COMPLETION_KEYS`). Docstrings are now
auto-spliced from `grok_web.schema.PARAMS` for all eight public methods,
keeping descriptions / defaults / CLI `--help` in lock-step with one source
of truth.

`download_video` also dropped the silent `parent_post_id` no-op — it had
been deprecated since the direct-lookup path landed and is now ignored
even on the legacy positional form (with a `DeprecationWarning`).

## Migration

The old positional/kwargs form keeps working through v0.19.x; you'll see a
single `DeprecationWarning` per call. To upgrade:

```python
# Before
await client.extend_video(video_id, prompt="continue", duration="6s")
await client.download_video(video_id, "out.mp4", prefer_hd=True)
await client.select_post(post_id, timeout=10.0)
await client.wait_for_video_completion(video_id, timeout=600)

# After
await client.extend_video({
    "video_id": video_id,
    "prompt": "continue",
    "duration": "6s",
})
await client.download_video({
    "video_id": video_id,
    "output_path": "out.mp4",
    "prefer_hd": True,
})
await client.select_post({"post_id": post_id, "timeout": 10.0})
await client.wait_for_video_completion({
    "video_id": video_id,
    "timeout": 600,
})
```

If you mix dict-style and kwargs (e.g. `extend_video({"video_id": ...},
prompt="...")`), the call now raises `TypeError` immediately rather than
silently dropping the kwargs — fail-fast over silent-wrong.

## Removal timeline

- **v0.19.0 (this release)**: dict-style is canonical. Positional form
  emits `DeprecationWarning`.
- **v0.20.0**: positional form removed. Callers that haven't migrated will
  get a clear `TypeError`.

That's roughly the same cycle as the `edit_image(post_id=, edit_prompt=)`
legacy keys (deprecated v0.18.0 → still accepted in v0.19.0 — same removal
window applies).
